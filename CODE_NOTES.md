## 1. Data Loading

**对应文件：** `src/carbon_portfolio/data.py`

### 模块作用

`data.py` 负责为后续组合优化准备可靠、格式统一的数据。

整体流程：

```text
请求价格数据
├─ 优先读取真实价格缓存或从 yfinance 下载
└─ 真实数据不可用时，按配置决定是否使用模拟 fallback
                         ↓
                   检查数据质量
                         ↓
                 计算简单日收益率
                         ↓
             交给优化和回测模块使用
```

这个模块不负责预测收益或配置组合，只负责保证输入数据可用，并清楚标识数据来源。

### `download_adjusted_close()`

#### 任务

优先从本地缓存取得公开历史 adjusted close；缓存不可用时，再从 Yahoo Finance 下载。
函数还负责统一单只和多只股票的数据结构，并在返回前完成质量检查。

#### 整体代码逻辑

1. **检查本地缓存**

   如果之前已经下载过数据，且没有要求强制刷新，就直接读取本地 CSV，避免反复联网。

2. **验证缓存是否适用**

   缓存读取后会交给 `validate_prices()`。该函数检查股票覆盖、日期索引、重复日期、数值
   格式和正价格。如果股票池改变导致缓存不完整，就放弃缓存，继续尝试下载。

3. **检查 `yfinance` 是否可用**

   只有真正需要联网下载时才导入 `yfinance`。如果依赖没有安装，函数会抛出清楚的
   `ImportError`，而不是在后续步骤产生难以理解的错误。

4. **从 Yahoo Finance 下载数据**

   缓存不可用时，通过 `yfinance` 下载指定股票的公开历史市场数据。

   下载参数明确设置 `auto_adjust=False`，然后由代码主动选择 `Adj Close`。这样可以清楚
   知道项目使用的是哪个价格字段，而不是依赖隐式调整。

5. **检查下载结果是否为空**

   如果 Yahoo Finance 没有返回任何观察值，函数会立即报错，不允许空数据继续进入收益率
   计算或回测。

6. **提取 adjusted close**

   Yahoo Finance 还会返回开盘价、最高价、成交量等字段，但项目只保留经过拆股和分红调整
   的收盘价。

7. **兼容单只和多只股票**

   `yfinance` 在不同下载情形下可能返回不同表格结构，所以函数分别处理单层列与多层列。

   - 多只股票通常返回 `MultiIndex`，代码从第一层字段中提取 `Adj Close`。
   - 单只股票可能返回普通列，代码保留 DataFrame 结构，并把列名改成 ticker。

8. **进行数据质量检查**

   确认数据非空、股票没有缺失、价格为正、日期有序且没有重复。

9. **保存并返回数据**

   将验证后的价格保存为本地缓存，再返回给收益率计算模块。

#### 流程概括

```text
检查真实数据缓存
├─ 缓存存在、无需刷新且验证通过 → 返回缓存价格
└─ 缓存不存在、要求刷新或验证失败
   ↓
检查 yfinance → 下载公开市场数据 → 提取 Adj Close
   ↓
验证价格
├─ 验证失败 → 报错
└─ 验证成功 → 保存缓存并返回
```

#### 研究含义

- 使用缓存提高可复现性，也减少同一实验反复下载造成的数据版本变化。
- 使用 adjusted close 是为了减少拆股和分红对收益率计算造成的机械扭曲。
- 缓存只能提高运行稳定性，不能证明数据具有机构级质量。

### `simulate_adjusted_close()`

#### 任务

在真实价格无法取得时，生成可复现、具有一定资产相关性的教学用模拟价格。它只用于测试
程序流程，不能替代真实市场数据，也不能支持投资、alpha 或气候影响结论。

#### 整体代码逻辑

1. **建立模拟日期**

   使用 `pd.bdate_range()` 生成工作日序列，并将结束日期排除在样本之外，使日期范围与
   yfinance 常见的 `end` 排他逻辑保持接近。

2. **检查样本长度**

   至少需要两个工作日，才能由价格计算一期收益率。如果日期范围太短，函数直接报错。

3. **初始化随机数生成器**

   使用固定 `seed` 创建 NumPy 随机数生成器。相同股票池、日期和 seed 会产生相同模拟
   路径，便于复现和自动测试。

4. **生成共同市场因子**

   为每个日期生成一个共同市场收益。所有股票都受到这个因子影响，因此模拟资产不会完全
   独立。

5. **为每只股票生成 beta**

   每只股票获得不同的市场敏感度，使共同市场冲击对不同资产产生不同幅度的影响。

6. **生成个股特有噪声**

   为每个日期和每只股票生成独立随机波动，用来表示共同市场因素无法解释的个股风险。

7. **组合模拟收益**

   将共同市场收益乘以各股票 beta，再加上个股噪声：

   $$
   r_{i,t} = \beta_i r_{m,t} + \epsilon_{i,t}
   $$

   其中：

   - `stock_return(i, t)`：股票 i 在日期 t 的模拟收益率；
   - `beta(i)`：股票 i 对共同市场因子的敏感度；
   - `market_return(t)`：日期 t 的共同市场收益；
   - `idiosyncratic_noise(i, t)`：股票 i 在日期 t 的个股特有噪声。

8. **限制极端日收益**

   使用 `np.clip()` 将模拟日收益限制在 -25% 到 25%，避免极端随机值产生非正价格或异常
   路径。这是程序稳定性处理，不是根据真实市场尾部风险校准的模型。

9. **把收益累积成价格**

   所有资产以 100 为起点，通过复合累计形成价格路径：

   $$
   P_{i,t} = 100 \prod_{s=1}^{t}(1+r_{i,s})
   $$

   也就是说，每一天的价格都建立在前一天累积结果之上，而不是把日收益率简单相加。

10. **返回标准价格表**

    将模拟矩阵转换为 DataFrame：日期作为索引，ticker 作为列名，使后续模块可以用同一
    接口处理真实和模拟数据。

#### 流程概括

```text
生成工作日日期
→ 固定随机种子
→ 生成共同市场因子
→ 生成各股票 beta
→ 生成个股噪声
→ 合成相关日收益
→ 限制极端收益
→ 复合为从 100 开始的模拟价格
```

#### 研究含义

- 共同市场因子使资产间出现正相关，更接近组合研究需要的数据结构。
- 固定随机种子保证的是“可复现”，不代表模拟过程“真实”。
- 模拟数据没有真实公司事件、行业结构、波动聚集、市场假期或尾部风险。

### `load_prices_with_fallback()`

#### 任务

提供一个统一、不会轻易中断的数据入口，同时明确区分真实数据与模拟数据。

#### 整体代码逻辑

1. **优先尝试真实数据**

   调用 `download_adjusted_close()`，读取缓存或从 `yfinance` 下载 adjusted close。

2. **真实数据成功时**

   返回价格表，并附带数据来源标签：

   ```text
   yfinance_adjusted_close
   ```

3. **真实数据失败时**

   记录原始错误，例如网络问题、`yfinance` 未安装、下载结果为空或股票缺失。

4. **检查是否允许 fallback**

   如果不允许使用模拟数据，程序直接停止并报告原始错误。

5. **明确警告用户**

   如果允许 fallback，程序会说明接下来使用模拟价格，而且模拟结果不能支持真实投资结论。

6. **优先读取模拟数据缓存**

   如果以前已经生成过模拟价格，就直接读取，保证结果可复现并减少重复生成。

7. **没有模拟缓存时生成数据**

   调用 `simulate_adjusted_close()` 创建模拟价格，并保存到带有 `SIMULATED` 标识的独立文件。

8. **返回数据来源标签**

   模拟数据会附带：

   ```text
   SIMULATED_FALLBACK
   ```

#### 流程概括

```text
尝试真实数据
├─ 成功 → 返回真实价格 + 真实数据标签
└─ 失败
   ├─ 不允许 fallback → 报错并停止
   └─ 允许 fallback → 读取或生成模拟价格 + 模拟标签
```

#### 研究含义

- 统一入口使后续代码不必分别处理真实和模拟价格。
- 数据来源标签是诚信控制，防止模拟结果被误认为真实回测。
- 使用广泛的 `except Exception` 能提高教学程序的连续性，但也可能捕获非网络类的程序
  错误。更严格的研究系统应捕获更具体的异常并记录正式日志。

### `validate_prices()`

#### 任务

把来源不同的价格表整理成统一、可检查、适合计算收益率的格式，并阻止明显无效的数据进入
后续研究。

#### 整体代码逻辑

1. **复制输入数据**

   创建价格表副本，避免清洗过程意外修改调用者保留的原始对象。

2. **统一日期索引**

   将索引转换成 Pandas 日期类型，确保排序、切片和滚动窗口使用真正的时间顺序。

3. **按日期升序排列**

   将最早日期放在前面。收益率必须按照正确时间方向计算，否则结果会失去含义。

4. **删除重复日期**

   同一天出现多条记录时，只保留第一条，避免相同日期被重复纳入收益率和协方差估计。

5. **统一为数值格式**

   尝试把所有价格转换为数值。不能转换的字符串或异常内容被标记为缺失值，而不是静默参与
   数学运算。

6. **检查股票池覆盖**

   如果调用者提供了目标 ticker 列表，函数会计算“要求的股票”与“实际返回列”的差集。
   缺少任何股票都会显式报错，不会悄悄缩小资产池。

7. **统一股票列顺序**

   按配置中的 ticker 顺序重新排列列，使价格、收益率、carbon intensity 和权重能够稳定对齐。

8. **检查是否存在可用数据**

   如果价格表为空，或者所有值都为空，函数立即报错。

9. **检查价格是否为正**

   adjusted close 必须严格大于零。零价格会造成无穷收益率，负价格不符合本项目的股票价格
   假设。

10. **返回清洗后的价格表**

    通过检查后，返回格式统一的数据供收益率计算使用。

#### 流程概括

```text
复制原始价格
→ 日期类型转换并排序
→ 删除重复日期
→ 转换为数值
→ 检查 ticker 是否完整
→ 按配置顺序排列股票
→ 检查非空和正价格
→ 返回标准价格表
```

#### 研究含义

- 不允许静默删除缺失股票，有助于保持不同运行之间的资产池可比性。
- 该函数没有填补部分缺失价格；缺失值会在收益率阶段继续处理。
- 它验证的是基础结构和明显错误，不是完整的数据供应商质量审计。

### `calculate_returns()`

#### 任务

把 adjusted close 转换为 mean-variance optimization 和历史回测所需的简单日收益率，并
控制缺失值处理方式。

#### 整体代码逻辑

1. **计算相邻日期的百分比变化**

   对每只股票分别计算：

   $$
   r_{i,t} = \frac{P_{i,t}}{P_{i,t-1}} - 1
   $$

   其中 `price(i, t)` 是股票 i 当日价格，`price(i, t-1)` 是其前一交易日价格。

   第一行没有前一期价格，因此第一期收益率自然为空。

2. **禁止自动填补价格**

   `fill_method=None` 明确要求 Pandas 不在计算收益率前自动填补缺失价格，避免人为制造零
   收益或错误跨期收益。

3. **处理无穷值**

   将正无穷和负无穷替换为缺失值。虽然价格验证已拒绝非正价格，这仍是一层防御性处理。

4. **根据参数删除不完整日期**

   - `drop_incomplete=True` 时，只要某只股票当日收益缺失，就删除整个日期，形成完整共同样本。
   - `drop_incomplete=False` 时，只有全部股票收益都缺失才删除该日期，部分缺失仍会保留。

5. **返回收益率矩阵**

   返回以日期为行、股票为列的收益率表，供 expected return、covariance matrix 和回测使用。

#### 流程概括

```text
Adjusted close
→ 计算简单日收益率
→ 不自动填补缺失价格
→ 将无穷值改为缺失值
→ 按配置删除不完整日期
→ 返回收益率矩阵
```

#### 研究含义

- Mean-variance optimization 使用收益率均值和协方差，而不是股票价格水平。
- 完整共同样本便于计算没有缺口的 covariance matrix，但删除日期也会减少样本。
- 当前实现没有使用 backward fill，因此不会把未来价格直接填回过去。

### 核心数学与金融概念

#### Adjusted close

Adjusted close 尝试修正拆股和分红的影响。若股票发生 2-for-1 split，未调整价格可能从
100 机械下降到 50，看起来像 -50% 收益；adjusted close 尝试消除这类非经济损失。

#### 简单收益率

$$
r_t = \frac{P_t}{P_{t-1}} - 1
$$

例如价格从 100 上升到 103，简单收益率为 3%。后续模块使用这些收益率估计：

$$
\mu = E[r], \qquad \Sigma = \operatorname{Cov}(r)
$$

- `mu` 表示各资产的历史平均收益率向量；
- `Sigma` 表示资产收益率之间的方差和协方差矩阵。

#### 共同市场因子

模拟数据采用简化的一因子结构：

$$
r_{i,t} = \beta_i r_{m,t} + \epsilon_{i,t}
$$

不同股票共享同一个 `market_return(t)`，因此资产收益之间会产生相关性；
`idiosyncratic_noise(i, t)` 表示个股特有波动。

#### Look-ahead bias

如果用后一天价格 backward fill 过去的缺失价格，就会在历史日期使用当时尚不可获得的
信息。当前代码不使用 backward fill，以减少这种明显的未来信息泄漏。

### 模块局限

- `yfinance` 是方便的公开数据源，不是机构级、point-in-time 数据库。
- Adjusted close 的质量和历史修订取决于第三方供应商。
- `pd.bdate_range()` 生成工作日，但没有排除所有美国证券市场假期。
- 固定的当前股票池存在 selection bias 和 survivorship bias。
- 删除任何股票缺失的日期可能减少样本，并形成完整案例选择。
- 缓存文件没有记录下载时间、供应商版本或正式数据 lineage。
- 模拟数据参数是人为设置的，没有校准真实波动、相关性或尾部风险。
- 模拟 fallback 只能证明代码流程可以运行，不能支持任何真实投资结论。

## 2. Carbon Intensity

**对应文件：** `src/carbon_portfolio/carbon.py`

### 模块作用

`carbon.py` 负责把公司碳披露整理成适合组合优化和历史回测使用的数据。

整体流程：

```text
读取年度碳强度数据
→ 检查数据字段、类型和计算口径
→ 根据调仓日期筛选当时已经公开的数据
→ 为每只股票选择最新可用披露
→ 对齐股票顺序
→ 计算组合加权碳强度
```

这个模块不决定组合权重，也不预测收益。它的核心任务是回答：

```text
在某个历史调仓日，我们当时能看到哪些公司的碳强度数据？
```

### `load_carbon_intensity()`

#### 任务

读取年度 carbon intensity CSV，将数据整理成统一格式，并阻止字段缺失、重复记录或计算
错误的数据进入回测。

#### 整体代码逻辑

1. **读取碳强度文件**

   使用 Pandas 读取 `data/carbon_intensity.csv`。每一行代表某家公司某个报告年度的排放、
   收入和碳强度记录。

2. **检查必需字段**

   确认文件包含 ticker、reporting year、available date、Scope 1、location-based
   Scope 2、revenue 和 carbon intensity 等核心字段。

   缺少任何字段都会显式报错，避免程序使用结构不完整的数据。

3. **复制数据**

   创建 DataFrame 副本，避免后续类型转换和筛选意外影响原始对象。

4. **统一 ticker 格式**

   把 `ticker` 转换为字符串，保证股票代码能够和价格表列名正确匹配。

5. **转换公开日期**

   把 `available_date` 转换为 Pandas 日期类型。

   这个字段表示排放和收入数据最早可以被历史投资者使用的时间，而不是排放发生的时间。

6. **转换数值字段**

   把报告年份、排放量、收入和碳强度转换为数值。无法转换的异常内容会直接报错，不会静默
   进入数学运算。

7. **检查重复记录**

   检查同一家公司、同一报告年度是否出现多条记录。如果存在重复记录，程序无法判断应使用
   哪一条，因此直接停止。

8. **检查有限数值**

   确认数值字段不包含 `NaN`、正无穷或负无穷。这些值会破坏碳暴露和优化约束计算。

9. **检查排放量**

   Scope 1 和 Scope 2 必须大于或等于零。本项目不单独处理碳移除、抵消或 avoided
   emissions，因此不接受负排放。

10. **检查收入**

    收入必须严格大于零。收入为零会导致碳强度无法计算；负收入也不符合当前简化定义。

11. **重新计算碳强度**

    程序根据原始字段重新计算：

    ```text
    carbon intensity = (Scope 1 + location-based Scope 2) / revenue in USD million
    ```

    然后将重新计算结果与 CSV 中保存的碳强度比较，用来发现录入错误、单位错误或组成部分
    与结果不一致的问题。

12. **检查股票池覆盖**

    如果调用者提供目标 ticker 列表，函数会检查每只股票是否至少拥有一条碳披露记录。缺少
    任何股票都会显式报错，不会自动缩小资产池。

13. **筛选目标股票**

    只保留当前项目股票池中的公司，避免无关记录进入后续处理。

14. **排序并返回**

    按照 `available_date → ticker → reporting_year` 排列，并重新生成连续行号，返回标准化
    年度碳数据面板。

#### 流程概括

```text
读取 CSV
→ 检查必需字段
→ 转换日期和数值类型
→ 检查重复记录
→ 检查有限值、排放和收入
→ 重新计算并验证碳强度
→ 检查股票池覆盖
→ 筛选、排序并返回数据
```

#### 研究含义

- 不直接信任 CSV 中的结果，而是用组成字段重新计算碳强度，有助于发现基础数据错误。
- 检查完整股票池覆盖，可以避免某些股票因为缺少碳数据而被悄悄排除。
- 这些检查只能发现明显结构问题，不能证明公开披露已经达到机构级数据质量。

### `carbon_snapshot_as_of()`

#### 任务

在指定历史日期，为股票池中的每家公司选择当时已经公开的最新一条 carbon intensity 记录。

这是 `carbon.py` 中避免 look-ahead bias 的核心函数。

#### 整体代码逻辑

1. **转换查询日期**

   把 `as_of` 转换为 Pandas 时间类型，确保可以与 `available_date` 正确比较。

2. **保存目标股票顺序**

   把 ticker 列表转换成普通列表。这个顺序之后会用于对齐价格、收益率、碳强度和组合权重。

3. **筛选当时已经公开的数据**

   只保留满足以下条件的记录：

   ```text
   available_date <= as_of date
   ```

   同时只保留当前股票池中的公司。报告年度早并不代表数据已经可用；真正决定能否进入回测
   的是 `available_date`。

4. **按公开时间排序**

   将可用记录按照 `available_date` 和 `reporting_year` 排序，使较新的记录排在后面。

5. **按公司分组**

   按 ticker 对数据分组。每家公司可能拥有多个报告年度，因此需要分别寻找最新记录。

6. **选择最新可用记录**

   每个 ticker 保留排序后的最后一条记录。

   例如在某个调仓日，公司已经公开 2019、2020 和 2021 年数据，但尚未公开 2022 年数据，
   函数会选择 2021 年记录。

7. **将 ticker 设为索引**

   把股票代码转换为索引，方便后续按股票顺序对齐。

8. **按目标股票池重新排列**

   使用配置中的 ticker 顺序重新排列结果，使碳数据顺序和收益率、权重保持一致。

9. **检查历史数据覆盖**

   检查每只股票是否在该历史日期拥有可用记录。如果某家公司当时还没有公开数据，函数会
   明确报错，而不会使用未来披露、平均值或随意填充。

10. **返回历史快照**

    返回指定日期下完整股票池的最新可用碳数据。

#### 流程概括

```text
接收历史日期和股票池
→ 筛选当时已公开的数据
→ 按公开日期排序
→ 按 ticker 分组
→ 每家公司选择最新记录
→ 按股票池顺序排列
→ 检查是否完整覆盖
→ 返回 point-in-time 碳数据快照
```

#### 研究含义

- 使用 `available_date` 而不是只使用 `reporting_year`，是减少碳数据 look-ahead bias 的
  关键。
- 在披露尚未覆盖全部股票前等待，可以避免把未来信息提前放入历史回测。
- 年度碳数据更新频率低，因此同一条披露可能在多个调仓日重复使用。

### `carbon_scores_as_of()`

#### 任务

把不同形式的碳数据统一转换成优化器需要的一维碳强度 Series。

#### 整体代码逻辑

1. **保存目标股票顺序**

   把传入的 ticker 转换为列表，作为最终输出顺序。

2. **识别静态 Series**

   如果输入是 Series，说明它是单元测试或教学小例子中的静态碳分数。函数会按目标股票池
   重新排列，并检查是否覆盖全部资产。

3. **识别真实年度面板**

   如果输入是 DataFrame，说明它是主程序使用的年度碳披露面板。函数调用
   `carbon_snapshot_as_of()`，选择指定日期已经公开的最新记录。

4. **提取碳强度列**

   从历史快照中只取 `carbon_intensity_tco2e_per_usd_mn`。

   优化器只需要每只股票的碳强度向量，不需要来源网址、报告年份等辅助字段。

5. **拒绝不支持的输入**

   如果输入既不是 Series 也不是 DataFrame，函数会报错。

6. **检查最终数值**

   确认所有碳强度不是缺失值、不是无穷值，且不小于零。

7. **设置输出名称**

   为 Series 设置清晰名称，便于输出表格和调试。

8. **返回碳强度向量**

   返回以 ticker 为索引的一维 Series，供优化器计算碳约束。

#### 流程概括

```text
接收碳数据
→ 判断 Series 或 DataFrame
→ 静态数据直接对齐
→ 年度面板按历史日期提取快照
→ 只保留碳强度
→ 检查完整性和有效性
→ 返回优化器需要的 Series
```

#### 研究含义

- 统一接口使 backtest 不需要关心碳数据是测试用 Series 还是正式年度面板。
- 主程序使用 DataFrame 面板，才能做到 point-in-time 的历史披露选择。
- Series 路径主要服务测试和教学，不应被误解为正式 carbon dataset。

### `portfolio_carbon_exposure()`

#### 任务

根据组合权重和每只股票的碳强度，计算组合加权平均碳强度。

#### 整体代码逻辑

1. **对齐碳强度顺序**

   按照 `weights.index` 重新排列碳强度。矩阵运算要求每个权重必须对应正确的股票。

2. **检查是否完整对齐**

   如果某个权重找不到对应的碳强度，函数立即报错。它不会忽略缺失股票，也不会把缺失值
   当作零。

3. **计算加权平均碳强度**

   通过权重和碳强度的点积计算：

   ```text
   portfolio carbon exposure = weights @ carbon intensity
   ```

   例如：

   ```text
   股票 A：权重 60%，碳强度 10
   股票 B：权重 40%，碳强度 50

   组合碳强度 = 0.6 * 10 + 0.4 * 50 = 26
   ```

4. **返回普通数值**

   将计算结果转换为 Python `float`，方便保存到回测诊断和绩效表中。

#### 流程概括

```text
接收权重和碳强度
→ 按权重中的股票顺序对齐
→ 检查每只股票是否都有数据
→ 计算权重与碳强度的点积
→ 返回组合加权碳强度
```

#### 研究含义

- 组合碳暴露是权重加权后的披露碳强度，不是组合实际排放量。
- 降低该指标不等于真实世界减排，也不代表投资产生了气候 impact。
- 该指标仍受公司披露口径、收入分母、Scope 边界和数据发布时间影响。

### 核心数学与金融概念

#### Carbon intensity

本项目使用的碳强度定义为：

```text
carbon intensity = (Scope 1 + location-based Scope 2) / revenue in USD million
```

它衡量单位收入对应的披露排放量，而不是公司的绝对排放量。

#### Point-in-time data

Point-in-time data 指在某个历史日期真实可获得的数据。即使某条记录对应 2022 年，如果
`available_date` 是 2023 年 8 月，那么它不能用于 2023 年 3 月的回测调仓。

#### Portfolio carbon exposure

组合碳暴露是权重和碳强度的点积：

```text
portfolio carbon exposure = sum(weight_i * carbon_intensity_i)
```

它是组合构建中的约束变量，不是收益预测信号。

#### Look-ahead bias

如果在历史回测中使用当时尚未公开的排放或收入数据，回测会隐含未来信息。当前模块通过
`available_date <= as_of date` 降低这种偏差。

### 模块局限

- 公开披露不等于机构级、审计一致或完全可比的数据集。
- 不同公司可能使用不同组织边界、Scope 2 方法、估算流程、重述政策和财年。
- 当前碳强度只包含 Scope 1 和 location-based Scope 2，没有纳入 Scope 3。
- 排放和收入报告期可能不完全匹配。
- 年度碳披露存在发布时间滞后，不能代表每日实时碳暴露。
- 收入下降可能导致碳强度上升，即使绝对排放没有增加。
- 组合加权碳强度下降不等于真实世界减排或投资 impact。

## 3. Portfolio Optimization

**对应文件：** `src/carbon_portfolio/optimization.py`

### 模块作用

`optimization.py` 负责在给定 expected returns、covariance matrix、risk aversion、weight constraints 和可选 carbon budget constraint 的情况下，求出组合权重。

整体流程：

```text
接收收益、风险和约束输入
→ 检查输入完整性与可行性
→ 构造 equal-weight benchmark
→ 构造 mean-variance objective
→ 加入基本约束：满仓、long-only、单股权重上限
→ 可选加入 carbon budget constraint
→ 调用 scipy SLSQP 优化器
→ 检查优化结果是否真的满足约束
→ 返回权重、预期收益、波动率、碳暴露和求解状态
```

这个模块不下载数据，也不做回测。它的核心问题是：

```text
在当前输入和约束下，每只股票应该配置多少权重？
```

---

### `OptimizationError`

#### 任务

定义一个专门用于组合优化失败、不可行或违反约束的错误类型。

#### 整体代码逻辑

1. **Carbon budget 不可行**

   即使把权重尽可能分配给低碳股票，也无法满足碳预算。

2. **优化器没有成功收敛**

   scipy SLSQP 返回失败状态。

3. **返回权重违反基本约束**

   例如权重和不接近 1，或出现明显负权重。

4. **返回权重超过 carbon budget**

   优化器声称成功，但结果实际违反碳约束。

#### 流程概括

```text
优化问题不可行或求解结果不可信
→ 抛出 OptimizationError
→ 阻止错误权重进入回测
```

#### 研究含义

在回测中，不能静默替换失败结果。如果某个调仓日优化失败，应该显式暴露问题，否则绩效结果会被污染。

---

### `OptimizationResult`

#### 任务

用一个结构化对象保存单次优化的结果。

#### 整体代码逻辑

1. **保存 portfolio name**

   记录当前结果属于 `equal_weight`、`mean_variance` 还是 `carbon_constrained`。

2. **保存 weights**

   保存每只股票的配置比例，这是优化模块最核心的输出。

3. **保存 expected return**

   根据输入的 expected returns 和最终权重，计算组合的预期收益。

4. **保存 volatility**

   根据 covariance matrix 和最终权重，计算组合的波动率。

5. **保存 carbon exposure**

   如果传入 carbon intensity，就计算组合加权碳强度。

6. **保存 success 和 message**

   记录优化是否成功，以及优化器返回的信息。

#### 流程概括

```text
优化器得到权重
→ 计算组合收益、风险和碳暴露
→ 打包成 OptimizationResult
→ 交给 backtest 和 reporting 使用
```

#### 研究含义

不要只保存权重。一个组合是否值得分析，还需要同时看收益、风险、碳暴露和优化状态。

---

### `equal_weights()`

#### 任务

生成 equal-weight portfolio，也就是每只股票配置相同权重。

#### 整体代码逻辑

1. **读取 ticker 数量**

   函数先计算资产池中有多少只股票。

2. **平均分配权重**

   每只股票权重为：

   ```text
   1 / 股票数量
   ```

   例如 10 只股票，每只股票权重就是 10%。

3. **返回 Pandas Series**

   用 ticker 作为索引，权重作为数值，保证后续可以和收益率、协方差、碳强度按股票名称对齐。

#### 流程概括

```text
输入股票列表
→ 计算股票数量
→ 每只股票分配相同权重
→ 返回 equal-weight 权重
```

#### 研究含义

Equal-weight 是本项目的基准组合。它不依赖 expected return，也不依赖 covariance matrix，因此不会受到收益预测误差或协方差估计误差的直接影响。

#### 局限

Equal-weight 虽然简单，但它并不代表“最优”。它忽略了股票之间风险差异、相关性差异、收益差异和碳强度差异。

---

### `_validate_inputs()`

#### 任务

在正式优化前，检查 expected returns、covariance matrix 和 carbon intensity 是否格式一致、数据完整、可以安全进入优化器。

#### 整体代码逻辑

1. **检查 expected returns 是否为空**

   如果没有任何预期收益输入，优化器没有目标可以比较不同组合。

2. **检查 covariance matrix 是否为空**

   如果没有协方差信息，优化器无法衡量组合风险。

3. **检查 covariance matrix 是否是方阵**

   covariance matrix 必须是 n × n。因为每只股票都要和自己以及其他股票形成一组风险关系。

4. **检查 expected returns 和 covariance matrix 的股票是否一致**

   收益向量和协方差矩阵必须描述同一组股票。

5. **统一股票顺序**

   按 expected returns 的顺序重排 covariance matrix，避免权重、收益和风险在股票维度上错位。

6. **转换为数值格式**

   把输入转成 float，避免字符串或异常类型参与数学计算。

7. **检查是否存在缺失值**

   如果 expected returns 或 covariance matrix 有缺失值，优化结果会不可靠。

8. **检查 covariance matrix 是否有限**

   避免无穷大或非数值进入优化器。

9. **检查 carbon intensity 是否对齐**

   如果提供 carbon intensity，就按同一股票顺序排列，并检查是否缺失。

#### 流程概括

```text
检查收益输入
→ 检查协方差输入
→ 对齐股票顺序
→ 转换数值类型
→ 检查缺失和异常值
→ 对齐 carbon intensity
→ 返回标准化输入
```

#### 研究含义

Portfolio optimization 对输入顺序非常敏感。如果 expected returns、covariance matrix、carbon intensity 和 weights 的股票顺序错位，优化器仍然可能运行，但结果会变成错误组合。

---

### `optimize_mean_variance()`

#### 任务

在 long-only、满仓、单股上限和可选 carbon budget 约束下，求解 mean-variance portfolio。

#### 整体代码逻辑

1. **先验证输入**

   调用 `_validate_inputs()`，确保收益、风险和碳强度数据可以进入优化。

2. **设定资产数量和初始权重**

   初始权重通常设为 equal-weight。优化器需要一个起点，之后再逐步寻找更优权重。

3. **设置单股权重上下界**

   每只股票权重下限为 0，上限为 `max_weight`。

   在本项目中，这代表：

   ```text
   不允许做空
   不允许单只股票权重过度集中
   ```

4. **设置满仓约束**

   所有权重之和必须等于 1：

   ```text
   sum(weights) = 1
   ```

   这表示资金全部投入股票池，不额外保留现金，也不使用杠杆。

5. **如果有 carbon budget，就加入碳约束**

   组合碳暴露必须小于或等于给定预算：

   ```text
   portfolio carbon exposure <= carbon budget
   ```

6. **先检查 carbon budget 是否可行**

   如果理论上的最低碳组合都无法满足 carbon budget，函数会直接报错，不让优化器在不可行问题上浪费时间。

7. **定义 objective function**

   本项目的目标函数可以理解为：

   ```text
   minimize risk penalty - expected return
   ```

   更具体地：

   ```text
   minimize 0.5 * risk_aversion * portfolio variance - portfolio expected return
   ```

   其中：

   ```text
   portfolio variance = weights' * covariance_matrix * weights
   portfolio expected return = expected_returns' * weights
   ```

8. **调用 scipy SLSQP 优化器**

   SLSQP 适合处理带有等式约束、不等式约束和权重边界的连续优化问题。

9. **检查优化器是否成功**

   如果优化器没有收敛，函数会抛出 `OptimizationError`。

10. **清理极小负数**

    数值优化有时会产生类似 `-0.0000000001` 的极小负权重。代码会把这类数值归零，并重新归一化。

11. **再次检查约束**

    即使优化器说成功，代码仍会检查：

    ```text
    权重和是否接近 1
    是否存在明显负权重
    是否超过 carbon budget
    ```

12. **计算组合结果**

    根据最终权重计算 expected return、volatility 和 carbon exposure。

13. **返回 OptimizationResult**

    把所有结果打包返回。

#### 流程概括

```text
验证输入
→ 设置初始权重
→ 设置权重边界
→ 设置满仓约束
→ 可选设置 carbon constraint
→ 定义 mean-variance objective
→ 调用 scipy optimizer
→ 检查求解状态
→ 检查约束是否真的满足
→ 返回结构化优化结果
```

#### 研究含义

这个函数体现了本项目最核心的研究问题：

```text
当我们要求组合既考虑收益和风险，又满足碳预算时，最优权重会如何变化？
```

#### 局限

- Expected return 由历史窗口估计，预测能力很有限。
- Covariance matrix 由历史样本估计，可能不稳定。
- SLSQP 是数值优化器，结果可能受参数、初始值和约束影响。
- 优化结果不是 alpha 证据，只是给定假设下的权重解。

---

### `_minimum_carbon_exposure()`

#### 任务

计算在当前股票池、long-only、满仓和单股上限约束下，理论上可以达到的最低组合碳暴露。

#### 整体代码逻辑

1. **按 carbon intensity 从低到高排序**

   低碳股票排在前面，高碳股票排在后面。

2. **从最低碳股票开始分配权重**

   尽可能先买低碳股票。

3. **每只股票最多分配 `max_weight`**

   即使某只股票碳强度最低，也不能超过单股权重上限。

4. **直到权重总和达到 1**

   权重分配满 100% 后停止。

5. **计算最低可行碳暴露**

   用这组“尽可能低碳”的权重计算组合碳暴露。

#### 流程概括

```text
按碳强度排序
→ 优先配置低碳股票
→ 每只股票最多配置 max_weight
→ 配满 100%
→ 得到当前约束下的最低碳暴露
```

#### 研究含义

这个函数不是在找收益风险最优组合，而是在回答另一个问题：

```text
给定股票池和单股权重上限，这个 carbon budget 有没有可能实现？
```

#### 例子

如果 carbon budget 设得比最低可行碳暴露还低，那么优化问题没有解。不是优化器能力问题，而是约束本身不可行。

---

### `_minimum_carbon_weights()`

#### 任务

生成一组尽可能降低组合碳暴露的权重。

#### 整体代码逻辑

1. **创建全零权重**

   一开始所有股票权重都是 0。

2. **按碳强度从低到高遍历股票**

   优先给低碳股票配置权重。

3. **每次分配不超过剩余权重和单股上限**

   如果还剩 40% 没分配，且单股上限是 25%，那么最多给当前股票 25%。

4. **更新剩余权重**

   每分配一只股票，就减少剩余可分配权重。

5. **直到组合权重分配满**

   当剩余权重接近 0 时，停止分配。

6. **检查是否成功分配满仓**

   如果因为股票数量太少或单股上限太低导致无法配满 100%，则抛出错误。

#### 流程概括

```text
初始化全零权重
→ 从最低碳股票开始
→ 按 max_weight 逐只分配
→ 直到权重总和为 1
→ 返回最低碳倾向权重
```

#### 研究含义

这个函数服务于 carbon budget 可行性检查。它帮助我们判断：

```text
在不做空、不加杠杆、单股有限制的前提下，最低能把组合碳强度压到哪里？
```

### 本模块中的核心金融逻辑

#### 1. 权重不是预测本身

优化器输出的是 portfolio weights，不是预测哪个股票一定会上涨。

#### 2. Expected return 只是输入假设

本项目使用历史窗口估计 expected return。它方便教学，但不是稳定的 alpha signal。

#### 3. Covariance matrix 描述共同波动

组合风险不仅取决于每只股票自己的波动，也取决于股票之间是否经常一起涨跌。

#### 4. Constraints 会改变最优解

满仓、long-only、单股上限和 carbon budget 不是附加说明，而是优化问题本身的一部分。

#### 5. Carbon constraint 是投资约束，不是收益预测信号

它限制组合加权碳强度，但不表示低碳股票未来一定表现更好。

#### 6. 优化失败本身也是重要结果

如果某个约束经常导致不可行或边界解，这说明模型设定值得重新检查。

---

### 一个小例子

假设只有三只股票：

```text
股票 A：expected return 高，carbon intensity 高
股票 B：expected return 中等，carbon intensity 中等
股票 C：expected return 低，carbon intensity 低
```

无碳约束 mean-variance portfolio 可能会偏向 A，因为 A 的预期收益较高。

加入 carbon budget 后，优化器必须控制组合平均碳强度，所以 A 的权重可能下降，B 或 C 的权重可能上升。

但这不意味着 A 一定被剔除，也不意味着 C 一定权重最高。最终权重同时取决于：

```text
expected return
covariance matrix
risk_aversion
max_weight
carbon intensity
carbon budget
```

所以 carbon-constrained portfolio 的意义不是“买最低碳股票”，而是在收益、风险和碳预算之间做一个受约束的权衡。
